## STEP 4: YAML ISSUE FORM SPECIFICATION

Every `.yml` issue form MUST follow this exact structure. No deviations.

```yaml
name: <Short human-readable name — shown in template chooser>
description: <One sentence — what this form is for>
title: "[<PREFIX>]: <short description>"
labels: [<label1>, <label2>]
assignees: []  # Leave empty unless specific maintainers detected
body:
  # Elements follow — see types below
```

### Valid Body Element Types:

**markdown** — Static informational text (no input, no id required)
```yaml
- type: markdown
  attributes:
    value: |
      Your message here. Supports **bold**, _italic_, [links](url).
```

**checkboxes** — For confirmations (duplicate check, CoC agreement, etc.)
```yaml
- type: checkboxes
  id: unique-id-here
  attributes:
    label: Label shown above checkboxes
    description: Optional helper text
    options:
      - label: The text of the checkbox item
        required: true  # or false
      - label: Another checkbox item
        required: false
```

**input** — Single-line text field
```yaml
- type: input
  id: unique-id-here
  attributes:
    label: Field Label
    description: Helper text shown below label
    placeholder: Example input text
  validations:
    required: true  # or false
```

**textarea** — Multi-line text field (for logs, descriptions, steps, etc.)
```yaml
- type: textarea
  id: unique-id-here
  attributes:
    label: Field Label
    description: Helper text
    placeholder: Detailed example text
    render: shell  # Optional: enables syntax highlighting. Values: shell, javascript, python, json, yaml, etc.
  validations:
    required: true  # or false
```

**dropdown** — Select from predefined options
```yaml
- type: dropdown
  id: unique-id-here
  attributes:
    label: Field Label
    description: Helper text
    multiple: false  # true allows multi-select
    options:
      - Option One
      - Option Two
      - Option Three
  validations:
    required: true  # or false
```

### ID Naming Rules:
- Lowercase, hyphen-separated
- Must be unique within the form
- No spaces, no underscores, no special characters
- Examples: `existing-issue-check`, `app-version`, `reproduction-steps`, `severity`

### Label Naming Rules:
- Labels referenced in `labels:` MUST match labels that exist in the repo
- Use only labels you will create in Step 7
- Multiple labels: `labels: [bug, needs-triage]`

## STEP 8: GITHUB ACTIONS WORKFLOWS

Generate only workflows that match detected project needs.

### Always generate: `.github/workflows/stale.yml`

```yaml
name: Mark Stale Issues and PRs

on:
  schedule:
    - cron: '0 0 * * *'  # Daily at midnight UTC
  workflow_dispatch:

permissions:
  issues: write
  pull-requests: write

jobs:
  stale:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/stale@v9
        with:
          repo-token: ${{ secrets.GITHUB_TOKEN }}
          stale-issue-message: >
            This issue has been automatically marked as stale because it has not had
            recent activity. It will be closed in 14 days if no further activity occurs.
            If this issue is still relevant, please leave a comment or add a label.
          stale-pr-message: >
            This pull request has been automatically marked as stale because it has not had
            recent activity. It will be closed in 14 days if no further activity occurs.
          close-issue-message: >
            This issue was closed automatically after being stale for 14 days.
            Please reopen it if it is still relevant.
          stale-issue-label: stale
          stale-pr-label: stale
          exempt-issue-labels: 'pinned,security,in-progress,confirmed'
          exempt-pr-labels: 'pinned,in-progress'
          days-before-stale: 60
          days-before-close: 14
```
